# 프로젝트 개발 요구명세서

## 1. 문서 정보
- 프로젝트명: 역전 회의 (회의 심판자) / MeetingReferee
- 문서 버전: v1.0
- 작성일: 2026-03-07
- 작성자: MeetingReferee 팀
- 상태: 초안(개발 착수용)

## 2. 프로젝트 개요
- 배경/문제: 회의에서 특정 인원의 발언 독점, 반복 논쟁, 논점 이탈이 발생하여 시간과 집중력이 소모됨.
- 목표: Deepgram 기반 실시간 화자 분리/발화 텍스트 분석으로 발언 지분을 시각화하고, 종료 시 회의 품질 지표를 제공함.
- 기대 효과:
  - 모든 참석자의 발언 기회 균형 유도
  - 비효율적 발언 패턴 조기 감지
  - 데이터 기반 회의 문화 개선

## 3. 범위
### 포함 범위 (In Scope)
- 실시간 음성 입력 수집 및 Deepgram STT/화자분리(Diarization) 연동
- 화자별 발화 텍스트/시간 집계
- 실시간 발언 지분(시간/비율) 대시보드
- 발언 독점/침묵 경고 룰 적용 및 UI 알림
- 회의 종료 후 품평 리포트 생성(요약 제외, 품질 지표 중심)

### 제외 범위 (Out of Scope)
- 회의 내용 요약/의사결정 자동 기록
- 개인 신상정보 기반 화자 신원 식별(실명 인증)
- 다국어 번역 기능
- 화상회의 플랫폼 자체 개발

## 4. 사용자 및 이해관계자
### 주요 사용자
- 회의 주최자(퍼실리테이터)
- 회의 참석자
- 팀 리더/매니저(사후 품질 검토)

### 이해관계자
- 제품 기획/개발팀
- 보안 및 컴플라이언스 담당
- 운영팀(로그/장애 대응)

## 5. 핵심 기능 요구사항
| ID | 기능명 | 설명 | 우선순위 (P0/P1/P2) | 수용 기준 |
|---|---|---|---|---|
| FR-001 | 실시간 음성 수집 | 마이크/회의 오디오 스트림을 실시간 수집한다. | P0 | 회의 시작 후 3초 이내 스트림 연결 완료 |
| FR-002 | Deepgram STT 연동 | 음성을 텍스트로 변환하고 화자 분리 결과를 수신한다. | P0 | 발화 단위로 `speaker_id`, `transcript`, `start/end time` 수신 |
| FR-003 | 화자별 텍스트 표기 | 화면에 화자 단위 발화 텍스트를 순차 표기한다. | P0 | 최근 발화가 2초 이내 UI에 반영 |
| FR-004 | 발언 지분 계산 | 화자별 발언 시간/횟수/비율을 실시간 계산한다. | P0 | 지표 오차율 5% 이내(샘플 검증 기준) |
| FR-005 | 발언 독점 감지 | 특정 화자 지분 과다 시 경고를 표시한다. | P0 | 기본 룰(예: 40% 초과 5분 지속) 충족 시 3초 이내 경고 |
| FR-006 | 침묵 참여자 감지 | 일정 시간 발언이 없는 참여자 비율을 표시한다. | P1 | 10분 이상 무발언 참여자 자동 표기 |
| FR-007 | 비효율 패턴 감지 | 장문 독백/반복/감정 고조 등 룰 기반 이벤트 감지 | P1 | 탐지 이벤트가 타임라인에 로그로 남음 |
| FR-008 | 종료 리포트 생성 | 회의 종료 시 품질 지표 리포트를 제공한다. | P0 | 종료 후 10초 이내 리포트 화면 생성 |
| FR-009 | 품평 메시지 자동 생성 | 지표 기반 피드백 문구를 생성한다. | P1 | 독점/균형/침묵 상태별 템플릿 메시지 출력 |
| FR-010 | 결과 내보내기 | 리포트를 PDF 또는 링크 형태로 공유한다. | P2 | 클릭 1회로 내보내기 수행, 실패 시 재시도 제공 |

## 6. 비기능 요구사항
### 성능
- STT 반영 지연: 평균 2초 이하, 최대 5초 이하
- 동시 회의 세션: 초기 20세션 이상 처리
- 60분 회의 기준 메모리 누수 없이 안정 실행

### 보안
- 전송 구간 TLS 적용
- 음성/텍스트 데이터 암호화 저장(저장 시)
- 역할 기반 접근제어(주최자/참석자/관리자)
- 민감 데이터 마스킹 정책 적용(선택)

### 가용성/안정성
- 월 가용성 99.5% 이상
- Deepgram 장애 시 재시도 및 상태 안내
- 네트워크 단절 시 로컬 버퍼링 후 복구 동기화

### 접근성/사용성
- 발언 지분은 색상+수치 동시 제공(색약 대응)
- 모바일/데스크톱 반응형 지원
- 경고 메시지는 방해 최소화(토스트/배지 중심)

## 7. 화면/UX 요구사항
- 핵심 사용자 흐름:
  - 회의 생성/입장 -> 실시간 모니터링 -> 경고 확인 -> 회의 종료 -> 품질 리포트 확인/공유
- 주요 화면 목록:
  - 회의 대기/입장 화면
  - 실시간 모니터링 대시보드(화자, 자막, 지분, 이벤트 타임라인)
  - 종료 리포트 화면(지분 차트, 품질 점수, 피드백 문구)
- 와이어프레임/디자인 링크: 추후 Figma 링크 연결

## 8. 데이터 요구사항
- 주요 엔티티:
  - Meeting(id, title, start_at, end_at, participants)
  - Utterance(id, meeting_id, speaker_id, transcript, duration, timestamp)
  - SpeakerStat(meeting_id, speaker_id, talk_time, talk_ratio, turn_count)
  - MeetingReport(meeting_id, score, warnings, feedback_message)
- 저장/조회 요구:
  - 실시간 세션 데이터는 메모리 캐시 우선, 종료 시 영속 저장
  - 리포트는 회의 종료 후 조회 가능
- 데이터 보존 정책:
  - 원본 음성: 기본 미저장(옵트인 시 저장)
  - 텍스트/지표: 기본 90일 보관 후 삭제

## 9. 외부 연동 요구사항
- 연동 대상: Deepgram API
- 방식(API/Webhook/Batch 등): 실시간 스트리밍 API(WebSocket)
- 실패 처리 정책:
  - 타임아웃/5xx 시 지수 백오프 재시도(최대 3회)
  - 연동 실패 시 "분석 불가" 상태 표시 및 로컬 로그 적재
  - 연속 실패 시 세션 운영자에게 경고

## 10. 제약사항 및 가정
- 기술 제약:
  - 화자 분리는 오디오 품질 및 발화 중첩에 따라 정확도 변동 가능
  - 브라우저 마이크 권한 필요
- 일정 제약: MVP 6주 내 출시 목표
- 예산/리소스 제약: Deepgram 사용량 기반 과금 한도 내 운영
- 가정:
  - 회의당 참여자 2~10명
  - 단일 언어(한국어) 우선 지원
  - 주최자는 회의 시작/종료 권한 보유

## 13. 수용 기준 (Acceptance Criteria)
- AC-001: 회의 중 화자별 발화 텍스트와 지분이 실시간으로 표시되어야 한다.
- AC-002: 특정 화자 독점 조건 충족 시 사용자에게 즉시 경고가 표시되어야 한다.
- AC-003: 회의 종료 후 10초 내 품질 리포트가 생성되어야 한다.
- AC-004: 리포트에는 최소 `화자별 지분`, `침묵 참여자`, `비효율 이벤트`가 포함되어야 한다.
- AC-005: 장애 발생 시 사용자에게 원인 상태(연동 실패/네트워크 지연)가 명시되어야 한다.
